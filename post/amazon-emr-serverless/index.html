Amazon EMR Serverless - Overview

Introduction

  1. What is Amazon EMR?

    • Amazon EMR is a cloud big data platform running large-scale distributed data processing jobs, interactive SQL queries, and machine learning (ML) applications using open-source analytics frameworks such as Apache Spark, Apache Hive, and Presto.
    • Understanding clusters and nodes
      • A cluster is a collection of Amazon Elastic Compute Cloud (Amazon EC2) instances. An instance in the cluster is called a node, and each node has role within the cluster, referred to as the node type.
      • The node types in Amazon EMR are as follows:
        • Master node: A node that manages the cluster by running software components to coordinate data distribution and tasks among other nodes for processing.
        • Core node: A node with software components that run tasks and store data in your cluster’s Hadoop Distributed File System (HDFS).
        • Task node: A node with software components that only runs tasks and does not store data in HDFS.

  2. What is Amazon EMR Serverless?

    • Amazon EMR Serverless is a serverless option on Amazon EMR that makes it easy to run open-source big data analytics frameworks without configuring, managing, and scaling clusters or servers.
    • Amazon EMR Serverless automatically provisions and scales the compute and memory resources required by your applications, and we only pay for the resources that the applications use.

  3. Amazon EMR Serverless Core concepts

    • Application: With EMR Serverless, we can create one or more applications (for example, Apache Spark or Apache Hive, or Presto) that use open-source analytics frameworks.
    • Job: A job is a request submitted to an EMR Serverless application that is asynchronously run and tracked through completion.
    • Workers: An EMR Serverless application internally uses workers to run your jobs. Depending on the open-source framework, EMR Serverless uses a default number of VCPU, memory, and local storage per worker, and we can override these defaults for your application.
    • Pre-initialized workers: Pre-initialized workers allow you to maintain a warm pool of workers for the application so that it can provide a sub-second response to start processing requests.

  4. How does the Amazon Ec2-based Data pipeline differ from Amazon EMR Serverless?

    • A typical pattern with data pipelines is to start a cluster, run a job, and stop the cluster when the job is complete. Because data is separated from computing, inputs and outputs for each job are persisted separately (for example, in Amazon S3).
    • When data sizes change over time, they have to resize clusters or use features like Amazon EMR managed to scale that automatically resize clusters.
    • EMR Serverless provides a more straightforward solution by eliminating the need for you to handle these scenarios. We don’t have to worry about instance selection, cluster sizes, cluster startup, cluster resize, stopping nodes, Availability Zone failover, or OS updates.

  5. What are the security best practices for Amazon EMR Serverless?

    • Amazon EMR Serverless provides several security features to consider as you develop and implement your security policies. The following best practices
    • Apply the principle of least privilege: EMR Serverless provides a granular access policy for applications using IAM roles, such as execution roles.
    • Isolate untrusted application code: EMR Serverless creates complete network isolation between jobs belonging to different EMR Serverless applications.
    • Role-based access control (RBAC) permissions: Administrators should strictly control Role-based access control (RBAC) permissions for EMR Serverless applications.

  6. Benefits:

    • Run frameworks more easily: Select the open-source framework you want to run for your application, such as Apache Spark, Hive, or Presto.
    • Scale on demand: Run analytics workloads at any scale with automatic on-demand scaling.
    • Optimize costs: In EMR Serverless, we pay only for what you use, and we can minimize concerns about over-or under-provisioning.

2. Submit Spark jobs from EMR Studio

  • Objective: In this demo, we will provision Amazon EMR serverless by creating an application and submitting an Apache Spark via EMR studio.

  • Below are the steps to perform this demo

    • Download the dataset for the demo, unzip it, and upload it to the Amazon S3 bucket provisioned by AWS Cloudformation in the previous step under the input folder.

    • Download the script for the demo and upload it to the Amazon S3 bucket provisioned by AWS Cloudformation in the previous step under the script folder.

    • In AWS Console, under services, search for EMR. In the EMR console, select EMR Serverless, or go to the EMR Serverless Console

    • Click Create and launch EMR Studio.

    • To create an EMR Serverless application, choose to Create application

    • Give a name to your application, my-serverless-application, as the name of our application. Choose Spark as the Type and emr-6.8.0 as the release version. Choose default settings. Click on Create application

    • You should see the Status Created.

    • Now you are ready to submit a job. To do this, choose to Submit the job.

    • In the Submit job screen, enter the details below:

      NameRuntime roleScript location S3 URIScript arguments
      word_countEMRServerlessS3RuntimeRoles3://YOUR_BUCKET/wordcount/scripts/wordcount.py[“s3://YOUR_BUCKET/wordcount/output/”]

      Replace YOUR_BUCKET with the S3 bucket name you noted in the Cloudformation Stack Outputs Section.

    • Click Submit job

    • You can see that the Application status shows as Starting, and the Job run status shows as Pending.

    • Once the job has been submitted, the Run status shows Success.

    • You can now verify that the job has written its output in the s3 path we provided as an argument when submitting it. You can go to the s3 path and see csv files successfully created by the EMR Serverless application.

3. Submit Hive jobs from EMR Studio

  • Objective: In this demo, we will submit a Hive job via EMR studio on an already existing Amazon EMR serverless cluster.

  • Below are the steps to perform this demo

    • Download the dataset for the demo and upload it to the Amazon S3 bucket provisioned by AWS Cloudformation in the previous step under the input folder.

    • Download create_taxi_trip.sql, count.sql and upload it to the Amazon S3 bucket provisioned by AWS Cloudformation in the previous step under the script folder.

    • On EMR Serverless Console, click on Create application. Enter the name hive-serverless, Select Type as Hive, and the latest release version.

    • You will notice the application is created and shows Starting.

    • Once the application is started, we can submit Hive jobs. Click on Submit job

    • In the Submit job screen, enter the details below:

    NameRuntime roleInitialization Script Location S3 URISScript location S3 URI
    Hive-Serverless-ConsoleServerlessS3RuntimeRoles3://YOUR_BUCKET/tripdata/scripts/create_taxi_trip.sqls3://YOUR_BUCKET/tripdata/scripts/count.sql

    Replace YOUR_BUCKET with the S3 bucket name you noted in the Cloudformation Stack Outputs Section.

    • Under Job configuration, update the DOC-EXAMPLE_BUCKET with the S3 bucket name you noted in the Cloudformation Stack Outputs Section.

    • Click Submit job

    • Once the job has been submitted, the Run status shows Success.

    • You can go over to Athena console to verify if the table got created and query it to check the data.

    • Click on the three dots and click Preview table
