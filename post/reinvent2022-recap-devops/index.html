Announcements from App Integration section

  1. AWS Step Functions launches large-scale parallel workflows for data processing

    • Overview

      • With Step Functions, we can orchestrate large-scale parallel workloads to perform tasks, such as on-demand semi-structured data processing. These parallel workloads let us concurrently process large-scale data sources stored in Amazon S3.
      • To set up a large-scale parallel workload in our workflows, include a Map state in Distributed mode. The Map state processes items in a dataset concurrently.
      • In Distributed mode, the Map state allows high-concurrency processing. In Distributed mode, the Map state processes the items in the dataset in iterations called child workflow executions.
      • In Distributed mode, we can run 10,000 parallel child workflow executions in parallel. Without Distributed mode, the Map state runs in the default Inline mode, which supports up to 40 concurrent iterations.

    • Consideration

      • One thing to be careful about is the concurrency limits of lambda functions in the account and the service limits of other services used in the execution flow.

  2. Amazon EventBridge Pipes is now generally available

    • Overview

      • EventBridge Pipes provides a more straightforward, consistent, and cost-effective way to create point-to-point integrations between event producers and consumers, expanding the EventBridge offering beyond event buses and scheduling.
      • EventBridge Pipes makes it easy to connect our applications with data from sources, including Amazon SQS, Amazon Kinesis, Amazon DynamoDB, Amazon Managed Streaming Kafka, self-managed Kafka, and Amazon MQ.
      • EventBridge Pipes supports the same target services as event buses, such as Amazon SQS, AWS Step Functions, Amazon Kinesis Data Streams, Amazon Kinesis Data Firehose, Amazon SNS, Amazon ECS, and event buses themselves.

    • How do Amazon EventBridge Pipes work?

      • Choose a source that is producing the events. Supported sources include: Amazon DynamoDB, Amazon Kinesis Data Streams, Amazon SQS, Amazon Managed Streaming for Apache Kafka, and Amazon MQ (both ActiveMQ and RabbitMQ).
      • (Optional) Specify an event filter to only process events that match our filter (weâ€™re not charged for events that are filtered out).
      • (Optional) Transform and enrich our events using built-in free transformations, AWS Lambda, AWS Step Functions, Amazon API Gateway, or EventBridge API Destinations to perform more advanced transformations and enrichments.
      • Choose a target destination from more than 14 AWS services, including Amazon Step Functions, Kinesis Data Streams, AWS Lambda, and third-party APIs using EventBridge API destinations.

    • Benefits

      • Visually compose serverless applications using AWS services with little guesswork.
      • Rapidly generate ready-to-deploy infrastructure as code (IaC) that follows best practices.
      • Maintain a model of our architecture with simplified team sharing and building.

Announcements from Security Compliance & Identity section

  1. Amazon Verified Permissions (Preview)

    • Overview

      • Amazon Verified Permissions, a scalable, fine-grained permissions management and authorization service for custom applications.
      • With Amazon Verified Permissions, application developers can let their end users manage permissions and share access to data.
      • This central fine-grained permissions management system simplifies changing and updating permission rules in a single place without needing to change the code.
      • Amazon Verified Permissions uses a custom policy language called Cedar to define fine-grained permissions for application users.
      • Access requests are evaluated in a few milliseconds, which allows continual verification as required by Zero Trust.
      • Amazon Verified Permissions can be used with any identity provider, such as Amazon Cognito.

    • Benefits

      • Accelerate application development by decoupling authorization from business logic.
      • Save time and resources with centralized permissions and policy lifecycle management.
      • Simplify compliance audits at scale using automated analysis to confirm that permissions work as intended.
      • Build applications that support Zero Trust architectures with dynamic real-time authorization decisions.

  2. Amazon Macie automated sensitive data discovery

    • Overview

      • An automated sensitive data discovery, a new capability in Amazon Macie that provides continual, cost-efficient, organization-wide visibility into where sensitive data resides across our Amazon Simple Storage Service (Amazon S3) estate.
      • With this new capability, Macie automatically and intelligently samples and analyzes objects across our S3 buckets, inspecting them for sensitive data such as personally identifiable information (PII), financial data, and AWS credentials.
      • Amazon Macie uses multiple automated techniques, including resource clustering by attributes such as bucket name, file types, and prefixes, to minimize the data scanning needed to uncover sensitive data in our S3 buckets.
      • Macie has multi-account support using AWS Organizations, allowing us to enable Macie across all our AWS accounts easier.
      • Macie applies machine learning and pattern-matching techniques to automatically identify and alert us to sensitive data, such as names, addresses, credit card numbers, or credential materials.

    • Features

      • With Macie, we can automate the discovery and reporting of sensitive data in two ways: by configuring Macie to perform automated sensitive data discovery, and by creating and running sensitive data discovery jobs.
      • To discover sensitive data with Macie, we can use built-in criteria and techniques, such as machine learning and pattern matching, to analyze objects in S3 buckets.
      • When we enable Macie, it automatically generates and begins maintaining a complete inventory of our S3 buckets. Macie also begins evaluating and monitoring the buckets for security and access control.
      • In Macie, a finding is a detailed report of sensitive data that Macie detects in an S3 object or a potential issue with the security or privacy of an S3 bucket.
      • Macie publishes findings to Amazon EventBridge as finding events to support integration with other services and systems.
      • If our AWS environment has multiple accounts, we can centrally manage Macie for accounts in our environment.
      • In addition to the Amazon Macie console, we can interact with Macie using the Amazon Macie API.

  3. Amazon Security Lake (Preview)

    • Overview

      • Amazon Security Lake automatically centralizes security data from the cloud, on-premises, and custom sources into a purpose-built data lake stored in our account
      • Security Lake makes it easier to analyze security data to understand our security across the entire organization.
      • Security Lake automatically gathers and manages all the security data across accounts and Regions.
      • Security Lake automates the collection of security-related log and event data from integrated AWS services and third-party services and manages the data lifecycle with customizable retention and replication settings.
      • Security Lake also converts ingested data into Apache Parquet format, and a standard open-source schema called the Open Cybersecurity Schema Framework (OCSF).

    • Benefits

      • Use the preferred analytics tools while retaining control and ownership of our security data.
      • Centralize data visibility from the cloud and on-premises sources across our accounts and AWS regions.
      • Normalize data to an open standard to share and use multiple analytics tools more efficiently.
      • Improve and manage security data for more efficient storage and query.

    • Features

      • Security Lake collects security logs and events from multiple AWS services.
      • Security Lake automatically partitions incoming data from natively-supported AWS services and converts it to a storage- and query-efficient Parquet format.
      • Subscribers consume data stored in Security Lake, and we specify that subscribers may consume data only from the sources and in the AWS regions.
      • We can centrally enable Security Lake across all available regions and multiple AWS accounts.
      • Security Lake is a configurable and customizable service. We can specify the sources, accounts, and regions we want to configure log collection.

  4. AWS KMS launches External Key Store

    • Overview

      • AWS Key Management Service (AWS KMS) introduces the External Key Store (XKS), a new feature for customers who want to protect their data with encryption keys stored in an external key management system under their control.
      • External key stores allow us to protect our AWS resources using cryptographic keys outside AWS.
      • An external key store is a custom key store backed by an external key manager that we own and manage outside of AWS.
      • The external key manager can be a physical or virtual hardware security module (HSMs) or any hardware-based or software-based system capable of generating and using cryptographic keys.
      • AWS KMS never interacts directly with the external key manager and cannot create, view, manage, or delete our keys. Instead, AWS KMS interacts only with the external key store proxy (XKS proxy) software we provide.

    • When should we opt for an external key store?

      • An external key store might be the right solution for our organization if we have the following requirements:
      • We are required to use cryptographic keys in our on-premises key manager or a key manager outside of AWS that we control.
      • We must demonstrate that our cryptographic keys are retained solely under our control outside the cloud.
      • We must encrypt and decrypt using cryptographic keys with independent authorization.
      • Key material must be subject to a secondary, independent audit path.

    • What are the changes in the Shared responsibility model?

      • With external key stores, the following principles change:
      • We are responsible for key operations’ availability, durability, and latency.
      • We are responsible for all costs for developing, purchasing, operating, and licensing the external key manager system.
      • We can implement independent authorization of all requests from AWS KMS to our external key store proxy.
      • We can monitor, audit, and log all operations of our external key store proxy and all operations of our external key manager related to AWS KMS requests.

  5. AWS Verified Access (preview)

    • Overview

      • AWS Verified Access allows us to provide secure network access without a VPN to corporate applications.
      • Verified Access verifies each access request in real-time and only connects users to the applications they are allowed to access, removing broad access to corporate applications and reducing the associated risks.

      • Verified Access also simplifies security operations by allowing administrators to set and monitor access policies efficiently, freeing time to update policies, respond to security and connectivity incidents, and audit compliance standards.

    • Benefits

      • Improved security posture - Verified Access evaluates each application access request in real-time. This makes it difficult for bad actors to move from one application to another.
      • Integration with security services â€“ Verified Access integrates with identity and device management services, including AWS and third-party services.
      • Improved user experience â€“ Verified Access removes the need for users to use a VPN to access our applications. This helps reduce the number of support cases arising from VPN-related issues.
      • Simplified troubleshooting and audits â€“ Verified Access logs all access attempts, providing centralized visibility into application access to help us quickly respond to security incidents and audit requests.

Announcements from Serverless section

  1. AWS Lambda SnapStart for Java functions

    • Overview

      • Lambda SnapStart for Java can improve startup performance for latency-sensitive applications by up to 10x at no extra cost, typically with no changes to the function code.
      • The most significant contributor to startup latency (often called cold start time) is Lambda’s time initializing the function, which includes loading the function’s code, starting the runtime, and initializing the function code. With SnapStart, Lambda initializes the function when we publish a function version.
      • Lambda takes a Firecracker microVM snapshot of the memory and disk state of the initialized execution environment, encrypts the snapshot, and caches it for low-latency access.
      • When we invoke the function version for the first time, and as the invocations scale up, Lambda resumes new execution environments from the cached snapshot instead of initializing them from scratch, improving startup latency.
      • Previously, the total cold invoke time was Init time + Duration but now, with SnapStart is Restore time + Duration. SnapStart has reduced the cold start duration from over 6 seconds to less than 200 ms.

    • Best Practice

      • Network connections
        • AWS recommends that we re-establish network connections in the function handler With SnapStart, a single snapshot is used as the initial state for multiple execution environments, and all execution environments return the same hostname value for InetAddress.getLocalHost(). For applications that require a unique execution environment identity or hostname value, AWS recommends that we generate a unique ID in the function handler.
        • AWS recommends that we avoid binding network connections to fixed-source ports
        • Lambda functions already cache DNS responses, so avoid using another DNS cache with SnapStart.
      • Performance tuning
        • To maximize the benefits of SnapStart, AWS recommends that we preload classes that contribute to startup latency in initialization code instead of in the function handler.
        • With this move, the latency associated with heavy class loading out of the invocation path, optimizing startup performance with SnapStart

    • Considerations and Limitations

      • SnapStart supports the Java 11 runtime.
      • SnapStart does not support provisioned concurrency, the arm64 architecture, the Lambda Extensions API, Amazon Elastic File System (Amazon EFS), AWS X-Ray, or ephemeral storage greater than 512 MB.
